== Introduction

Account Ledger is a Spring Boot application that serves as a ledger system for the fictional bank "Comptes En Ligne" (CEL). It maintains a source of truth for account balances and provides functionality for customers to:

1. View their accounts
2. Check account balances
3. View account movements (transactions)

The application is built using Spring Boot and follows Behavior-Driven Development (BDD) practices with Cucumber for testing and Cukedoctor for documentation generation.

=== Domain Model

The core domain concepts in the application are:

* *Account*: Represents a bank account with an IBAN, customer, label, and creation timestamp
* *Customer*: Represents a bank customer with ID, name, and contact information
* *Movement*: Represents a transaction on an account with timestamp, amount, and currency
* *Money*: Represents a monetary amount with a specific currency and provides currency conversion

== Prerequisites

* Java 21 or higher
* Maven 3.8 or higher

== Building and Running

=== Building the Project

[source,bash]
----
# Clean and build the project
mvn clean install

# Build without running tests
mvn clean install -DskipTests
----

=== Running the Application

[source,bash]
----
# Run using Maven
mvn spring-boot:run

# Run the JAR file directly
java -jar target/account-ledger-by-junie-1.0-SNAPSHOT-exec.jar
----

== Testing

=== Testing Framework

The project uses Cucumber for BDD testing with Spring Boot integration. Key components:

* *Cucumber*: For writing and running BDD tests
* *Spring Boot Test*: For integration testing with Spring Boot
* *JUnit 5*: As the underlying test runner
* *Cukedoctor*: For generating living documentation in French

=== Running Tests

[source,bash]
----
# Run all tests
mvn test

# Run a specific test class
mvn test -Dtest=SimpleTest

# Run Cucumber features
mvn verify
----

NOTE: Cucumber tests must be run through the `LedgerFeatures` test runner, not directly from feature files. The test runner is configured to find all feature files in the `src/test/resources/features` directory.

=== Adding New Tests

==== JUnit Tests

1. Create a new test class in the `src/test/java/io/github/rdlopes/ledger/test` directory
2. Annotate with `@SpringBootTest` for integration tests
3. Use standard JUnit 5 annotations (`@Test`, etc.)

Example:
[source,java]
----
package io.github.rdlopes.ledger.test;

import org.junit.jupiter.api.Test;
import org.springframework.boot.test.context.SpringBootTest;

import static org.junit.jupiter.api.Assertions.assertTrue;

@SpringBootTest
public class SimpleTest {

    @Test
    public void contextLoads() {
        // This test will pass if the Spring context loads successfully
        assertTrue(true, "Spring context loaded successfully");
    }
}
----

==== Cucumber Tests

1. Create a feature file in `src/test/resources/features` directory
2. Write scenarios using Gherkin syntax in French language
3. Create step definitions in `src/test/java/io/github/rdlopes/ledger/features/steps` directory

The project supports writing feature files in French. Use the following annotations for step definitions:

* `@Etantdonné` or `@Etantdonnéque` for "Given" steps
* `@Lorsque` for "When" steps
* `@Alors` for "Then" steps

== Documentation Generation

The project uses Cukedoctor to generate documentation from Cucumber features:

[source,bash]
----
# Generate documentation
mvn cukedoctor:execute
----

The generated documentation will be available in `target/cukedoctor/index.html`.

== Project Structure

* `src/main/java`: Main application code
** `io.github.rdlopes.ledger.application`: Application configuration and entry point
** `io.github.rdlopes.ledger.domain`: Domain model classes
** `io.github.rdlopes.ledger.service`: Service classes implementing business logic
* `src/main/resources`: Application configuration
* `src/test/java`: Test code
** `io.github.rdlopes.ledger.features`: Cucumber test configuration and step definitions
** `io.github.rdlopes.ledger.test`: JUnit tests
* `src/test/resources`: Test resources
** `features`: Cucumber feature files
** `cukedoctor-intro.adoc`: Introduction for the generated documentation
* `target`: Build output
** `cukedoctor`: Generated documentation

* `src/test/resources/application-features.yaml` for tests

The project uses the following Spring profiles:
